THE PROBLEM I LIVE EVERY MONTH

Reconciling money
isn't an OCR problem.
It's a trust problem.

My spending is scattered across three card apps, a bank CSV, UPI screenshots, and a wallet of paper receipts. The same coffee shows up twice with two different amounts. A receipt gets double-counted. OCR quietly turns ₹450 into ₹480.

The hard part was never reading the documents. It's trusting the merged result. That needs a system — one that extracts in parallel, verifies its own work, links duplicates across sources, and quarantines anything it can't prove instead of silently corrupting the ledger.

“Tracking expenses across UPI apps is a nightmare.” — Build at Damco's own example problem. This is the system answer to it.

WHAT BREAKS — AND THE GUARDRAIL FOR IT

        
When unsure, it quarantines.
It never corrupts.

        
Damco asks for honesty about what's broken. Here's what naturally fails, and the named guard that catches it.

      

      

        
F1Vision misreads a digit
Amount extracted twice; if the reads disagree → quarantine.

        
F2Double-counted purchase
Fuzzy match across sources collapses duplicates to one.

        
F3Cross-source conflict
₹450 vs ₹540 → anomaly → both quarantined with evidence.

        
F5Bank CSV schema drift
Pandera contract + fuzzy header recovery; self-heals or quarantines.

        
F6Model rate-limit / 5xx
Retry with backoff + jitter, then degrade — the run always completes.

        
F8Dashboard connects late
Decoupled execution + per-run replay buffer; or it polls the result.

      

      

        What I'd build next, honestly: confidence calibration against a larger labeled set ·
        an active-learning loop where every human quarantine-resolution becomes a new eval row ·
        a durable Postgres checkpointer + SQS so runs resume across restarts.
